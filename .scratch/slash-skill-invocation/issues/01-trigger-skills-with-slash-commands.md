# Trigger skills with slash commands in chat and issue comments

Status: ready-for-human
Category: enhancement

## Summary

Users should be able to invoke available agent skills directly from Multica text inputs by typing `/`, similar to slash commands in chat tools.

The first target surfaces are:

- Chat input
- Issue comment composer

First implementation should include both chat input and issue comment composer.

Chat support must avoid schema migrations in the first slice. The server can use the already-available user chat message content on the claimed task to detect `skill://<id>` invocations for that execution.

When the user types `/`, Multica should show the skills available in the current workspace and let the user select one to insert or invoke.

## Problem

Today, skills are available to the coding agent harness, but they are not discoverable or invokable from inside the product workflow. This makes skill usage depend on out-of-band knowledge: the user must know that a skill exists and how to ask for it.

Inside an issue or chat thread, the natural intent is often contextual: “run triage here”, “turn this into issues”, “grill me on this plan”. Slash-triggered skill discovery would make those actions visible at the point of need.

## Proposed behavior

- In supported rich/text inputs, typing `/` opens a command menu.
- The menu lists skills available to the current workspace.
- Each item shows at least:
  - Skill name
  - Short description
- Selecting a skill inserts a structured invocation into the composer or triggers an invocation flow.
- Invocation should preserve surrounding context:
  - In chat: current chat/workspace context
  - In issue comments: issue title, body, comments, and workspace context

## Initial acceptance criteria

- A user can type `/` in the chat composer and see available workspace skills.
- A user can type `/` in an issue comment composer and see available workspace skills.
- Skill results are filtered by the typed query after `/`.
- Selecting a skill makes the intended invocation explicit before execution.
- The UI handles the empty state when no skills are available.
- The implementation does not hardcode the skill list in the view layer; available skills come from core/API data or a platform adapter.

## Decisions so far

- Selecting a `/skill` inserts an explicit invocation; it does not execute immediately.
- Skills are workspace-scoped.
- In issue comments, the invocation is persisted as a visible comment.
- Slash commands in this feature are only for skills, not a generic command palette.

## Resolved design

### Scope

Enable slash-triggered skill invocation in conversation composers:

- Chat input
- New issue comment composer
- Issue comment reply composer

Do not enable it in issue descriptions, existing comment edit mode, agent instructions, or other document-style editors in the first slice.

### Stored syntax

Selecting a skill inserts a structured Markdown link followed by a trailing space:

```md
[/triage](skill://<skill-id>) 
```

The user can keep typing freeform instructions after the link:

```md
[/diagnose](skill://<skill-id>) investigate why desktop reloads after deleting a workspace
```

The full comment/message remains visible and user-authored. The link is the durable marker the backend can parse.

### Slash menu behavior

- The menu lists all workspace skills.
- Each item shows skill name and short description.
- Filtering uses the typed query after `/`.
- Trigger only when `/` is at the start of text or preceded by whitespace.
- Do not trigger inside URLs or paths like `https://x/y` or `foo/bar`.
- Typing whitespace after `/` or `/query` without selecting a menu item dismisses the menu and keeps the literal text.
- Never auto-insert a skill; insertion happens only on explicit selection.
- Keyboard/mouse support:
  - ArrowUp/ArrowDown navigate
  - Enter selects highlighted item
  - Escape closes
  - Click selects

### Execution semantics

Skill invocation does not create a special execution path. It augments normal task execution only when the invocation is in the content that triggered that task.

- Issue comment task: parse skill invocations from the triggering comment content (`trigger_comment_id` / `TriggerCommentContent`).
- Chat task: parse skill invocations from the user chat message content already present on the claimed task (`ChatMessage`). Avoid schema migrations in this first slice.
- Assignment-triggered issue task: do not scan historical comments for skill invocations.
- Historical invocations in older comments/messages must not affect future executions.
- Unassigned issues may still receive comments containing skill invocations. No blocking and no warning. If no task is triggered, the invocation remains a normal visible comment.

### Skill injection

For a task with explicit skill invocations, inject:

```text
agent attached skills ∪ explicitly invoked workspace skills
```

Explicit invocation does not permanently attach the skill to the agent and does not modify agent configuration.

If a skill is both attached and explicitly invoked, dedupe it by skill ID/name before sending to the daemon.

When a task includes explicitly invoked skills, the agent prompt/context should say so, for example:

```md
The user explicitly invoked these skills for this task:
- triage
```

### Permissions

Any workspace member who can comment or chat can invoke any visible workspace skill via `/`. Invocation does not edit the skill or attach it permanently to an agent.

3. Define the persisted representation in issue comments/chat messages.
